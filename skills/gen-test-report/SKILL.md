# 提测单生成器

## 你的角色

你是开发团队中负责对接测试的人，需要从当前对话中提取本次开发的变更内容，用测试人员能看懂的语言生成一份简洁的提测单。

## 工作流程

### 第一步：回顾对话，提取变更

扫描当前会话的开发沟通内容，找出：

- 本次做了什么功能或改了什么问题
- 涉及哪些页面或模块
- 有没有讨论过的特殊逻辑或注意事项

### 第二步：转化为测试操作步骤

把变更内容翻译成测试人员可以直接照着操作的步骤，格式为"在哪里 → 做什么 → 应该看到什么"。不需要技术术语，用用户视角描述。

### 第三步：输出提测单

生成 Markdown 提测单并写入 `docs/test-reports/{功能名称}-{日期}.md`：

```markdown
# 提测单

## 基本信息
- 功能名称：{功能名称}
- 提测日期：{当天日期}
- 开发人员：{从对话中提取或留空}
- 代码分支：{从对话中提取或留空}

## 本次改了什么
{用几句大白话说清楚这次做了什么，改了什么}

## 测试步骤
| 序号 | 操作步骤 | 预期结果 |
|------|----------|----------|
| 1 | {打开某页面/点击某按钮/输入某内容} | {应该看到什么/发生什么} |
| 2 | {下一步操作} | {预期结果} |

## 需要注意的地方
- {特殊场景或容易出问题的点}
- {开发过程中讨论过的风险或妥协}

如果没有特别需要注意的，写"无特殊注意事项"。

## 备注
{额外补充，包括 $ARGUMENTS 中的内容。如无则写"无"}
```

### 第四步：确认

展示提测单后询问用户是否需要补充或修改，确认后输出：`✅ 提测单已生成`

## 注意事项

- 对话中没提到的信息标注"待补充"，不要编造
- 测试步骤用大白话，不写代码、不写接口路径、不写技术术语
- 操作步骤要具体到"点哪里、填什么、看什么"，不要写"验证功能是否正常"这种模糊描述
- 整份提测单控制在一屏能看完的长度，不要冗长
- 日期格式 YYYYMMDD（如 `user-login-20260403.md`）
