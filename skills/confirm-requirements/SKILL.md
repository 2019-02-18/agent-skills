# Skill: confirm-requirements

> **需求确认 + Mermaid 流程图 · Requirements Confirmation Flowchart**

---

## 适用场景 / When to Use

在你开始任何功能开发、组件重构或 Bug 修复之前，使用本 Skill 让 AI 主动对齐需求、消除歧义，并输出可视化流程图作为双方共同认可的"开发契约"。

Use this skill before starting any feature, refactor, or bug fix to ensure the AI aligns with your intent, eliminates ambiguity, and produces a visual flowchart as a shared "development contract".

---

## 激活方式 / Activation

在对话中直接说：

```
请先确认需求 / Please confirm requirements first
```

或将本文件加入系统提示词，AI 将在每次收到编码请求时**自动触发**需求确认流程。

---

## AI 行为规范 / Agent Behaviour

当收到功能开发或修改请求时，**必须**按以下顺序执行，不得跳过：

### Step 1 — 解析请求，识别模糊点

仔细阅读用户的请求，列出所有不确定项，分为以下四类：

| 类别 | 示例问题 |
|---|---|
| **UI / 交互行为** | 点击按钮后是跳转页面还是弹出 Modal？过渡动画是否需要？ |
| **数据与接口** | 数据从哪个接口获取？字段名是什么？分页还是滚动加载？ |
| **边界与异常** | 空数据时显示什么？接口超时如何处理？最大字符数限制？ |
| **验收标准** | 哪些场景算通过？是否需要兼容移动端/旧浏览器？ |

### Step 2 — 追问（≤ 5 个核心问题）

用**有序列表**提出问题，每个问题附上可选答案（便于用户快速回复）：

```
在开始开发之前，我需要确认以下几点：

1. **跳转方式**：点击"查看详情"后，是在当前页面展开内容，还是跳转到新页面？
   - A) 当前页面展开（Accordion）
   - B) 跳转到 /detail/:id
   - C) 新标签页打开

2. **加载状态**：数据请求期间是否需要骨架屏（Skeleton）？
   - A) 需要骨架屏
   - B) 普通 Loading Spinner 即可
   - C) 无需加载态

3. **空数据状态**：列表为空时显示什么？
   - A) 显示"暂无数据"文案
   - B) 显示插图 + 引导文案
   - C) 隐藏整个区块

（如有其他特殊要求请补充说明）
```

### Step 3 — 等待用户回复

**不得在用户回复之前开始写代码。** 如果用户明确表示"直接开始"或"按你的理解来"，则采用最常见的业界实践作为默认值，并在流程图中注明。

### Step 4 — 生成需求确认流程图

收到回复后，将完整需求整理为 Mermaid 流程图：

````markdown
```mermaid
flowchart TD
    A([用户触发：点击"查看详情"]) --> B{当前页面还是新页面？}
    B -- 当前页面 --> C[展开 Accordion 组件]
    B -- 新页面 --> D[路由跳转 /detail/:id]
    C --> E[发起 GET /api/item/:id]
    D --> E
    E --> F{请求状态}
    F -- 加载中 --> G[显示 Skeleton 骨架屏]
    F -- 成功 200 --> H[渲染详情内容]
    F -- 失败 / 超时 --> I[显示错误 Toast + 重试按钮]
    H --> J{数据为空？}
    J -- 是 --> K[显示空状态插图 + 引导文案]
    J -- 否 --> L([渲染完成 ✓])
```
````

### Step 5 — 请求确认

在流程图下方附上：

```
以上流程图是否准确反映了你的需求？
✅ 确认无误，开始开发
✏️  需要调整：[请说明修改点]
```

**只有在用户确认后，才开始编写代码。**

---

## 流程图规范 / Flowchart Conventions

| 节点类型 | 用途 | Mermaid 语法 |
|---|---|---|
| 圆角矩形 `([...])` | 开始 / 结束节点 | `A([开始])` |
| 菱形 `{...}` | 判断 / 条件分支 | `B{已登录？}` |
| 矩形 `[...]` | 操作 / 处理步骤 | `C[跳转 /login]` |
| 平行四边形 `[/... /]` | 用户输入 / 输出 | `D[/输入手机号/]` |

- 每个分支必须有标签（`-- 是 -->` / `-- 否 -->`）
- 节点文字使用**用户语言**（中文请求则用中文）
- 接口调用节点标注 HTTP 方法和路径，如 `GET /api/users`
- 错误处理分支不可省略

## 完整示例 / Full Example

**用户输入：**
> 帮我做一个用户登录页面，支持手机号+验证码登录。

**AI 输出（Step 2）：**

```
在开始开发之前，我需要确认以下几点：

1. **登录成功后的跳转**：登录成功后跳转到哪个页面？
   - A) 跳转到首页 /
   - B) 跳转到上一个来源页（redirect 参数）
   - C) 由你指定路径

2. **验证码有效期**：验证码有效期是多少分钟？超时后如何提示？
   - A) 5 分钟，超时显示"验证码已过期，请重新获取"
   - B) 10 分钟
   - C) 由后端控制，前端无需处理

3. **发送频率限制**：点击"获取验证码"后，倒计时几秒才能再次发送？
   - A) 60 秒
   - B) 30 秒
   - C) 无限制

4. **错误处理**：手机号未注册时，是提示"手机号不存在"还是统一提示"验证码错误"（更安全）？
   - A) 区分提示（用户体验更好）
   - B) 统一提示（安全性更高）
```

**用户回复后，AI 输出（Step 4）：**

```mermaid
flowchart TD
    A([用户访问 /login]) --> B[/输入手机号/]
    B --> C[点击"获取验证码"]
    C --> D[POST /api/auth/send-otp]
    D --> E{发送成功？}
    E -- 是 --> F[开始 60s 倒计时，禁用按钮]
    E -- 否 --> G[Toast: 发送失败，请稍后重试]
    F --> H[/输入 6 位验证码/]
    H --> I[点击"登录"]
    I --> J[POST /api/auth/verify-otp]
    J --> K{验证结果}
    K -- 成功 200 --> L{有 redirect 参数？}
    K -- 验证码错误/过期 --> M[Toast: 验证码错误或已过期]
    K -- 请求失败 --> N[Toast: 网络异常，请重试]
    L -- 是 --> O[跳转到 redirect 页面]
    L -- 否 --> P([跳转到首页 /])
    M --> H
```

## 注意事项 / Notes

- 本 Skill 适用于**功能开发类**请求；对于简单的 CSS 修改或一行代码修复，可酌情简化确认步骤
- 流程图应聚焦**主流程 + 关键异常分支**，避免过度细化导致图形过于复杂
- 如果用户在同一对话中已经确认过相同需求，无需重复追问，直接引用已确认内容
